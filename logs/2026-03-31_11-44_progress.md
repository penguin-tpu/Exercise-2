# Progress Log

## Timestamp

2026-03-31 11:44:55 PDT

## Current Progress

- Implemented the first accelerator-side tensor/vector placeholder slice on top of the scalar RV32I and machine-trap baseline.
- Added tensor register scoreboarding through the existing engine issue path so tensor producers and consumers respect the same event-scheduled visibility model as scalar operations.
- Implemented tensor load/store, vector add, and MXU matmul semantics using the Torch backend plus the existing vector/MXU/load-store units.
- Added builder helpers so synthetic accelerator programs can emit tensor operations without bypassing the current `Program` and `Instruction` architecture.
- Marked the current `INSTRUCTIONS.md` item as started, as requested.

## Current Implementation Status

- New accelerator-side instructions now modeled:
  - `tload`
  - `tstore`
  - `vadd`
  - `matmul`
- These instructions:
  - issue through the same engine/event queue as scalar instructions
  - use tensor-register scoreboarding for RAW/WAW hazards
  - use load-store, vector, and MXU unit occupancy appropriately
  - rely on Torch-backed tensor payloads while keeping control flow and timing in normal Python state
- Verification is green with the new tensor slice included:
  - `python3 -m compileall perf_modeling tests`
  - `uv run python -m unittest tests.test_rv32i tests.test_rv32i_engine tests.test_tensor_slice`

## Current Roadblocks

- Remote push remains blocked by the current environment’s network restriction, so new commits can be made locally but may fail to publish.
- The tensor slice is currently synthetic-program driven; it is not yet reachable from RV32I binary decode because these are accelerator-specific placeholder ops, not RISC-V encodings.
- The environment still emits a Torch warning because NumPy is absent from the `uv` environment.

## Current Questions

- Whether the next slice should add tensor/vector load-store builders and microbenchmarks only, or start defining a more explicit accelerator ISA encoding path.
- Whether DMA should be brought into the same vertical slice next, so tensor movement is no longer modeled only through the shared load-store path.

## Pending Design Decisions

- How to represent future accelerator instructions relative to the RV32I frontend:
  - custom decoded opcodes only
  - coprocessor-style instructions
  - mixed-program bundles
- Whether tensor memory movement should stay scalar-addressed or shift toward descriptor-based operands as the model grows.

## Next-Step Plan

- Add focused microbenchmarks and more builder coverage for mixed DRAM/scratchpad tensor movement.
- Decide the first durable accelerator-side instruction representation beyond the current placeholder opcodes.
