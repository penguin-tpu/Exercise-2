# Progress Log

## Timestamp

2026-03-31 03:01:30 PDT

## Current Progress

- Implemented the first working RV32I execution slice on top of the existing combined functional/performance scaffold.
- Replaced the placeholder decoder with a real RV32I machine-code decoder and a minimal ELF32 loader for RISC-V loadable segments.
- Wired the engine issue path, event scheduling, register scoreboarding, control-flow fetch stalls, and unit occupancy through the existing event queue model.
- Added a usable CLI entrypoint in `main.py` that can execute a raw RV32I binary or ELF image.
- Added focused execution tests for decode breadth, scalar ALU and branch flow, DRAM load/store behavior, timing stalls, and ELF loading.

## Current Implementation Status

- The simulator can now load raw binary or ELF32 RV32I images, populate memory, start from the image entry point, and execute through halt.
- Supported RV32I functionality currently includes:
  - U-type ops: `lui`, `auipc`
  - Jumps: `jal`, `jalr`
  - Branches: `beq`, `bne`, `blt`, `bge`, `bltu`, `bgeu`
  - Loads: `lb`, `lh`, `lw`, `lbu`, `lhu`
  - Stores: `sb`, `sh`, `sw`
  - ALU-immediate ops: `addi`, `slti`, `sltiu`, `xori`, `ori`, `andi`, `slli`, `srli`, `srai`
  - ALU-register ops: `add`, `sub`, `sll`, `slt`, `sltu`, `xor`, `srl`, `sra`, `or`, `and`
  - System/fence slice: `fence`, `ecall`, `ebreak`
- Architectural results become visible on completion events, not issue, and dependent instructions stall on the scoreboard until writeback.
- `python3 -m compileall perf_modeling tests` passes.
- `uv run python -m unittest tests.test_rv32i tests.test_rv32i_engine` passes.

## Current Roadblocks

- The current milestone is scalar-only; vector/tensor architectural state remains scaffold-level.
- The ELF loader currently handles the subset needed for ELF32 little-endian RV32I bare-metal images and does not yet model relocations or richer section semantics.
- Torch imports successfully under `uv`, but it emits a warning because NumPy is not installed in the environment.

## Current Questions

- Whether the next slice should prioritize CSR/trap semantics for a more complete privileged or semihosted RV32 environment, or accelerator-specific tensor/vector instructions.
- Whether scratchpad should remain a fixed memory-mapped window or move to a richer interconnect/memory-map abstraction before vector/tensor instructions land.

## Pending Design Decisions

- Exact memory map beyond the current flat DRAM plus fixed scratchpad aperture.
- Whether to model fences as a broader memory-ordering mechanism once DMA/vector/tensor traffic is issuing concurrently.
- How far to push ELF support before bringing in a dedicated parser dependency.

## Next-Step Plan

- Add CSR/trap coverage and a more explicit machine-exception path where bare-metal workloads need it.
- Extend the same event-scheduled execution flow to vector/tensor placeholder instructions once the scalar baseline is stable.
- Add more end-to-end binary tests, including loops, function-call patterns, and mixed DRAM/scratchpad accesses.
